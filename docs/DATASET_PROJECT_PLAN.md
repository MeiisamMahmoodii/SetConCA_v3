# Dataset Project Plan: Latent-Set Data For SetConCA And Intervention-Tested Steering

Date: 2026-05-13

## 0. Paper Framing

The core contribution should be the dataset and representation benchmark. Steering should be treated as a second-stage application until the behavioral evidence is strong enough.

Use these terms consistently:

| Avoid | Prefer |
| --- | --- |
| same area under the surface | source-certified shared latent variable |
| proof of meaning | operational certificate of latent-set validity |
| causal steering dataset | contrastive intervention benchmark |
| causal steering direction | intervention-tested steering direction |

This keeps the claims defensible. A hidden-state intervention is causal in the narrow sense that we intervene on the model computation, but it does not automatically prove that the learned direction is a human-clean causal concept.

## 1. Why The Current Dataset Is Not Enough

The current V2 dataset is useful, but it mostly tests one relation:

```text
same source sentence -> constrained rewrites
```

That is a good controlled start. It forces lexical diversity and supports set-size experiments. But it still has two weaknesses for the next scientific claim:

1. The set identity is anchored to a single original sentence, so the model may still learn "paraphrase of this exact sentence" rather than a deeper latent concept.
2. The causal steering branch currently depends on manually reviewed topic-like concepts from AG News. This can produce useful directions, but it does not yet give a broad, reusable benchmark for alignment and steering.

The next dataset should treat each set as samples from a shared latent variable, not merely rewritten surface text.

## 2. Target Output

Build two linked datasets from the same source registry:

### A. SetConCA latent-set dataset

A JSONL dataset where each row is a set:

```json
{

  "latent_type": "answer_entity",
  "latent_key": "Arthur's Magazine",
  "source_dataset": "hotpotqa/hotpot_qa",
  "domain": "wikipedia_qa",
  "texts": [
    {"text": "Which magazine was started first Arthur's Magazine or First for Women?", "view_type": "question"},
    {"text": "Arthur's Magazine began publication before First for Women.", "view_type": "claim"},

  ],
  "evidence": [
    {"kind": "supporting_fact", "title": "Arthur's Magazine", "sent_id": 0}
  ],
  "positive_edges": [["0", "1"], ["1", "2"]],
  "negative_pool_ids": ["..."],

    "lexical_diversity_ok": true,
    "embedding_cluster_ok": true,
    "nli_or_evidence_ok": true,
    "graph_connectivity_ok": true
  }

```

This is the direct replacement or extension for `sets.jsonl`.

### B. Contrastive intervention benchmark dataset

A paired/contrastive dataset where each latent variable has:

- positive prompts/responses that express the target behavior or concept,
- negative or neutral controls,
- hard negatives from nearby concepts,
- evaluation prompts that are held out from training,
- automatic metrics plus a manual-audit path.

Example row:

```json
{
  "behavior_id": "truthfulness_misconception",
  "axis": "truthful_vs_misleading",
  "train_positive": ["A correct answer that resists the misconception."],
  "train_negative": ["A common but false answer."],
  "eval_prompts": ["Question where the misconception is tempting."],
  "scorers": ["exact_answer", "llm_judge", "refusal_or_uncertainty"],
  "capability_controls": ["same topic factual QA without misconception pressure"]
}
```

## 3. Dataset Sources To Use

The best plan is not one dataset. It is a source registry with several latent-set generators.

| Source | Use | Why It Helps | Main Risk |

| FEVER / FEVEROUS | Claim/evidence sets | Claims are tied to Wikipedia evidence and labels. Good for truthfulness, support/refute, and evidence steering. | Some claims are generated by perturbation; need filter for naturalness. |
| SNLI / MultiNLI / AllNLI | Entailment neighborhoods | Human-written entailments, neutrals, contradictions. Good for formal positive/negative edges. | NLI artifacts; should not be the only source. |
| Quora duplicates | Intent/question-equivalence sets | Graph connected components of duplicate questions create naturally varied wording. | Duplicate labels can be noisy and often shallow. |
| PAWS | Hard lexical controls | High word-overlap non-paraphrases prevent lexical shortcuts. | Pair format; needs graph/set construction. |

| RealToxicityPrompts | Toxicity steering/eval | Prompt beginnings with toxicity annotations. | Use only for benchmark/eval; handle harmful content carefully. |
| BBQ | Bias steering/eval | Ambiguous and disambiguated QA over nine social dimensions. | Sensitive social-bias content; needs clear safety policy and reporting. |
| SycophancyEval / Anthropic evals | Sycophancy steering/eval | Free-form tasks for agreement/flattery behavior. | Some data is benchmark-like; keep held out from training. |

### Recommended source order

Do not start with all sources. The first paper-quality dataset should start narrow:

1. FEVER / FEVEROUS for claim-evidence fact latents.
2. HotpotQA for answer/evidence multi-hop latents.
3. AllNLI or SNLI/MultiNLI for entailment/event-scene latents.
4. PAWS as lexical-adversarial hard negatives, not a main positive-set source.

Quora, TruthfulQA, BBQ, RealToxicityPrompts, sycophancy data, and algorithmic/code/math data should enter after the latent-set schema and validation reports are stable. They are useful, but they add more noise, safety review, and evaluation complexity.

## 3.1 Latent-Type Taxonomy

Do not treat every source as if it has the same kind of latent variable. Each accepted set must declare its latent type.

| Latent type | Example | Main positive evidence | Main evaluation |
| --- | --- | --- | --- |
| Entity latent | same answer entity | answer id, alias-normalized entity key, support facts | retrieval, clustering |
| Fact latent | same supported/refuted claim | claim id, evidence page/sentence, label | evidence consistency, hard-negative separation |
| Intent latent | duplicate questions | duplicate graph component | intent retrieval |
| Event/scene latent | NLI premise cluster | premise/hypothesis links, entailment labels | entailment/contradiction separation |
| Behavior latent | truthfulness, sycophancy resistance | contrastive behavior pairs | intervention benchmark |
| Algorithmic latent | same rule or operation | generated logic/math/code template id | reasoning generalization |

The first release should focus on entity, fact, and event/scene latents. Behavior latents should stay in the intervention benchmark. Algorithmic latents are optional but scientifically valuable as a small non-natural-language pilot.

## 4. The Mathematical Criterion For "Same Area Under The Surface"

We cannot prove true human meaning from text alone. But we can build a rigorous operational proof that a set is sampled from the same latent source under explicit assumptions.

Let each text view be an observed variable:

```text
X_i = g_i(Z, E_i)
```

where:

- `Z` is the latent object we want SetConCA to learn: answer entity, claim, event, relation, intent, behavior, or evidence cluster.
- `g_i` is the view-generation process: question writing, evidence sentence, claim writing, duplicate question, entailment, summary, etc.
- `E_i` is view-specific noise: wording, genre, length, author, style, or prompt format.

A valid set is not defined by surface similarity. It is defined by evidence that all `X_i` share the same `Z`.

### Acceptance rule

For a candidate set `S = {x_1 ... x_n}`, require:

1. Shared latent key:
   `latent_key(x_i) = z` for every member, coming from a source annotation such as answer id, evidence page, duplicate component, claim id, or support fact.

2. Graph connectivity:
   Build a graph where nodes are texts and edges are source-certified positive relations: duplicate, entailment, same answer, same evidence page/sentence, same claim support, same QA answer. Accept only if the positive graph is connected.

3. Non-trivial surface diversity:
   Pairwise lexical overlap must not be too high on average. Use Jaccard/content-token overlap, character n-gram overlap, and length/style diversity.

4. Semantic consistency:
   A strong independent validator should agree. Use embedding clustering plus NLI/evidence checks. Existing `src/setconca_v2/semantic_validation.py` can become one validator, but the new pipeline needs source-specific validators too.

5. Hard-negative separation:
   For each set, sample nearby negatives that share topic words or entity type but have a different latent key. A set is stronger if validators separate positives from hard negatives.

### Proof sketch

Assume the source annotation process has precision `p` for the latent relation, independent validators have false positive rates `q_1 ... q_k`, and hard-negative mining estimates topic confounding. If a set passes source-key equality, connected positive graph validation, independent semantic validation, and hard-negative separation, then the probability that all views are only surface-related but do not share `Z` is bounded by:

```text
P(false latent set) <= (1 - p) * product(q_j) * c
```

where `c` is a confounding factor estimated from hard-negative error. This is not metaphysical proof of meaning. It is a measurable dataset certificate: the set is accepted only when multiple independent mechanisms agree that a shared latent variable exists.

For reporting, each set should carry a `latent_set_confidence` score:

```text
confidence = source_precision_weight
           + graph_connectivity_weight
           + validator_margin_weight
           + hard_negative_margin_weight
           + diversity_weight
           - leakage_penalty
```

The weights must not be hand-waved. Add a calibration step before large-scale building:

1. Sample 100 candidate sets across FEVER, HotpotQA, and NLI.
2. Have 2-3 annotators independently rate whether each is a valid latent set.
3. Measure agreement and adjudicate disagreements.
4. Fit or grid-search the confidence weights against this small gold set.
5. Freeze the scoring config for the pilot release and report it in the dataset card.

## 5. Set Builders

### Builder 1: Evidence-grounded QA sets

Sources: HotpotQA, Natural Questions if added later, TriviaQA if licensing/use fits.

Latent variable:

- answer entity,
- answer relation,
- supporting fact bundle,
- multi-hop bridge.

Views:

- original question,
- answer statement,
- supporting evidence sentence,
- generated explanation grounded in evidence,
- counterfactual/hard-negative question with same topic but different answer.

Why this is strong:

The set members can be lexically different while tied by answer and evidence. HotpotQA specifically includes multi-document support and sentence-level supporting facts.

### Builder 2: Claim/evidence/truth sets

Sources: FEVER, FEVEROUS.

Latent variable:

- claim truth relation to evidence,
- supported/refuted fact,
- evidence page or evidence sentence.

Views:

- claim,
- evidence sentence,
- neutral restatement,
- support/refute answer,
- hard negative claim from same page but different fact.

Use for:

- SetConCA latent facts,
- truthfulness steering,
- hallucination suppression,
- evidence-use benchmark.

### Builder 3: Entailment component sets

Sources: SNLI, MultiNLI, AllNLI.

Latent variable:

- premise scene/event,
- entailment cluster,
- contradiction axis.

Views:

- premise,
- multiple entailments,
- neutral hard negatives,
- contradictions.

Use:

- clean positive/negative training edges,
- calibrating validator thresholds,
- studying whether SetConCA separates entailment from lexical overlap.

### Builder 4: Duplicate-intent graph sets

Sources: Quora duplicate questions, AskUbuntu duplicate questions through MTEB if useful.

Latent variable:

- user intent.

Process:

1. Build graph nodes as questions.
2. Add duplicate edges.
3. Use connected components as sets.
4. Remove weak components with low edge density or high semantic disagreement.
5. Add hard negatives from same topic/entity but different component.

Use:

- realistic casual-language variation,
- steering directions for user intent, uncertainty, help-seeking, recommendation, troubleshooting.

Status: future work for the first release. This builder is useful, but it should not block the core dataset because duplicate labels can be noisy and component-level latent keys are weaker than evidence-backed QA or claim data.

### Builder 5: Adversarial lexical-control sets

Sources: PAWS plus generated controls.

Latent variable:

- paraphrase vs non-paraphrase under high lexical overlap.

Use:

- not mainly for positive sets,
- for hard negatives and leakage checks.

This directly attacks the failure mode where a model learns word overlap instead of latent relation.

### Builder 6: Algorithmic latent pilot

Sources: generated logic puzzles, arithmetic word problems, simple code snippets, or symbolic templates.

Latent variable:

- same underlying operation,
- same logical relation,
- same program behavior,
- same math transformation.

Use:

- small optional pilot only,
- tests whether SetConCA can recover shared structure beyond linguistic topic and genre,
- helpful as a reviewer-facing stress test if the NLP sources look too language-specific.

Status: future work unless the first three NLP builders are already stable. This should not be part of the minimum viable benchmark.

### Most challenging builder

The duplicate-intent graph builder is likely the hardest to validate. Quora-like duplicate edges are useful because they contain casual natural language, but the latent key is weaker than FEVER evidence or HotpotQA support facts. Connected components can merge related but not identical intents, and noisy edges can poison a whole component. This builder should come after FEVER and HotpotQA, not before them.

## 6. Contrastive Intervention Benchmark Design

The steering data should not just be categories. It should be contrastive and intervention-testable.

Each steering axis needs four splits:

1. `train_contrast`: examples used to derive a steering direction.
2. `dev_calibration`: examples used to choose layer, alpha, sign, and prompt style.
3. `eval_behavior`: held-out prompts used for final behavioral claims.
4. `capability_control`: prompts checking whether the intervention breaks general ability.

Recommended first axes:

| Axis | Positive | Negative | Source |
| --- | --- | --- | --- |
| Truthfulness | correct answer, uncertainty when needed | popular misconception or hallucination | TruthfulQA, FEVER |
| Evidence use | answer grounded in evidence | unsupported assertion | FEVER, HotpotQA |
| Sycophancy resistance | keeps correct answer under pressure | agrees with false user premise | SycophancyEval, generated from QA |
| Toxicity reduction | safe continuation | toxic continuation | RealToxicityPrompts |
| Bias resistance | unknown or evidence-based answer | stereotype-driven answer | BBQ |
| Refusal calibration | refuses harmful request | over-complies with harmful request | safety eval sources, carefully curated |
| Helpfulness style | concise useful answer | evasive/vague answer | generated plus human audit |

For the first strong paper claim, prefer fewer axes with more data. Start with:

1. truthfulness / misconception resistance,
2. evidence use / unsupported assertion,
3. sycophancy resistance.

Use FEVER and HotpotQA latent sets to generate additional contrast pairs for truthfulness and evidence-use, then keep held-out evaluation rows source-disjoint from steering-direction training.

For each axis, store:

- positive and negative exemplars,
- hard negatives,
- prompt templates,
- answer keys,
- scorer type,
- topic/domain labels,
- risk category,
- whether safe to use for training or eval only.

## 7. Benchmark Design

The benchmark should evaluate both SetConCA representation learning and steering.

### A. Representation benchmark

Metrics:

- set retrieval: given one view, retrieve other views from the same latent set,
- clustering: recover latent set ids from activations,
- positive-vs-hard-negative margin,
- cross-model bridge transfer using current `real_minus_shuffled_topk`,
- set-size scaling from `S=2` to `S=16`,
- cross-domain generalization: train on QA/evidence, test on duplicate-intent or NLI sets.

Baselines:

- pointwise TopK baseline already in the repo,
- mean activation direction,
- CAA-style contrast vector,
- sentence embedding clustering,
- random/shuffled latent keys,
- lexical overlap baseline.

Required ablations:

| Ablation | Purpose |
| --- | --- |
| remove hard negatives | measures whether negatives drive the gain |
| remove source graph validation | measures whether graph construction matters |
| remove NLI/evidence validator | measures semantic-filter contribution |
| shuffle latent keys | checks whether gains depend on real sets |
| use only constrained paraphrases | compares against current V2 dataset |
| use only one source | tests cross-source generalization |

### B. Intervention benchmark

Metrics:

- target behavior gain,
- opposite-sign suppression,
- random-direction control,
- nearest-topic hard-negative leakage,
- capability retention,
- fluency/perplexity or judge-rated coherence,
- prompt sensitivity,
- layer/alpha stability,
- transfer across model family.

A steering result is claim-worthy only if:

1. active sign improves held-out target behavior,
2. opposite sign reduces or reverses the effect,
3. random direction does not match the gain,
4. nearby hard-negative behavior does not rise as much,
5. capability controls do not collapse,
6. the result repeats across at least two prompt templates or seeds.

## 7.2 Model Evaluation Report

The model report should answer different questions:

- Does SetConCA beat pointwise and lexical baselines?
- Does it improve with set size?
- Does it transfer across source domains and model families?
- Does it resist lexical-adversarial traps?
- Are gains still present under shuffled-key and hard-negative controls?
- Do activation-derived concept signatures stay stable across split halves?
- Do thresholded variable-size dictionaries transfer across model families?
- Do intervention-tested directions change held-out behavior without breaking capability controls?

## 7.3 Activation-Consistency Validation

Source evidence and text validators are necessary, but the final question for SetConCA is representational: does a candidate set produce a stable shared activation signature?

Dimension counting alone is not enough. If a model has hidden size around 4096 and a set has more than 8000 samples, linear dependencies are guaranteed, but that does not prove a real concept. The useful test is whether a shared direction or subspace generalizes to held-out members and beats hard negatives.

Validation procedure:

1. Collect activations for many views from a candidate latent set.
2. Split the views into two or more random halves.
3. Estimate a shared signature from each split:
   - centroid direction,
   - PCA subspace,
   - SetConCA sparse code support,
   - thresholded concept-feature set.
4. Measure split-half stability:
   - cosine between centroid directions,
   - principal-angle similarity between subspaces,
   - overlap of thresholded feature supports.
5. Test held-out positives against hard negatives:

```text
activation_validation_margin =
    mean(score(signature, held_out_positive))
  - mean(score(signature, hard_negative))
```

A set is stronger if:

- directions from different splits agree,
- held-out positives score higher than hard negatives,
- same-topic and lexical-adversarial negatives do not activate the signature,
- the signature is stable across view types, not just one genre.

This becomes an additional validator, not a replacement for source evidence.

## 7.4 Thresholded Concept Dictionary

Fixed TopK can be misleading. Some concepts may be clean and sparse; others may need more features; some may have no stable features and should be rejected.

Instead of forcing every concept to have exactly `K` active features, build a variable-size dictionary entry:

```text
concept C -> all features that pass trust thresholds
```

Candidate thresholds:

- positive mean activation above calibrated threshold,
- positive activation frequency above threshold,
- positive-vs-hard-negative margin above threshold,
- split-half stability above threshold,
- view-type stability above threshold.

Prefer calibrated or normalized thresholds over raw values such as `0.8`, because activation scales differ across models, layers, and training runs. Useful options:

- z-score threshold,
- percentile threshold,
- positive-negative margin,
- calibrated post-LayerNorm score,
- held-out validation threshold from the 100-set manual calibration round.

Example dictionary entry:

```json
{
  "concept_id": "fever_fact_001",
  "model": "llama3",
  "layer": "60pct",
  "active_features": [12, 91, 305],
  "scores": {
    "12": {"margin": 0.44, "split_stability": 0.81},
    "91": {"margin": 0.39, "split_stability": 0.74},
    "305": {"margin": 0.33, "split_stability": 0.68}
  }
}
```

Some concepts may keep 2 features, some 20, and some 0. A zero-feature result is useful evidence that the set is not clean or that the chosen layer/model does not represent it cleanly.

## 7.5 Cross-Family Dictionary Alignment

Different LLM families may represent the same concept with different hidden sizes, sparsity, and feature counts. Therefore, cross-model evaluation must compare concept subspaces or functional behavior, not exact feature ids.

Planned experiment:

1. Train SetConCA separately on Llama activations.
2. Train SetConCA separately on Qwen activations.
3. Build thresholded variable-size dictionaries for each model.
4. Learn linear bridges between the learned concept spaces:
   - Procrustes,
   - ridge regression,
   - CCA,
   - optional optimal transport,
   - optional small MLP as a nonlinear diagnostic, not the main claim.
5. Evaluate on held-out concepts:
   - map Llama concept signatures into Qwen space,
   - retrieve the matching Qwen concept,
   - score target positives above hard negatives,
   - compare against shuffled labels and random bridges.

The key question is not:

```text
Does Qwen use the same number of features as Llama?
```

The key question is:

```text
Does the mapped Llama concept recover the same held-out latent set in Qwen
better than hard negatives and shuffled controls?
```

Metrics:

- Top-1 / Top-5 concept retrieval,
- positive-vs-hard-negative activation margin,
- split-half stability,
- `real_minus_shuffled` transfer score,
- subspace overlap / principal-angle similarity,
- dictionary efficiency: number of features needed for the same score.

This directly tests whether separately trained SetConCA dictionaries can align across different model families. It is more ambitious than within-family size transfer and should be framed carefully as cross-family linear recoverability, not proof of identical internal representations.

Add scripts:

```text

scripts/build_steering_axes.py

```

Start with FEVER, HotpotQA, NLI, and PAWS controls. Keep Quora and behavior/safety sources for future work unless the core dataset is already passing validation gates.

Revised first implementation order:

1. FEVER builder: easiest sanity check for claim/evidence schema.
2. HotpotQA builder: richer multi-hop/entity/evidence structure.
3. NLI builder: event/scene and entailment/contradiction controls, with artifact checks.
4. PAWS hard-negative adapter.
5. Stop and evaluate. Only add Quora, behavior, toxicity/bias, or algorithmic builders if the core benchmark is already passing validation gates.

Expect high rejection rates. To get 2,000 accepted sets, budget 10,000-16,000 candidate sets, especially once hard-negative and view-balance filters are active.

### Phase 3: Validators

Extend the existing semantic validator into a multi-signal validator:

- source annotation check,
- graph connectedness,
- embedding centroid/margin,
- NLI consistency,
- lexical diversity,
- hard-negative separation,
- leakage check against labels and obvious keywords.
- genre/style shortcut check,
- source-only classifier check,
- dynamic adversarial negative check for high-confidence sets.

Output rejected rows too, exactly like the current `attempts.jsonl` audit trail.

Validation should be tiered for cost:

1. Cheap filters: schema validity, source id, view-type count, lexical overlap, length, duplicate text.
2. Medium filters: embedding centroid/margin, nearest-neighbor hard negatives, genre classifier.
3. Expensive filters: NLI, LLM judge, dynamic adversarial negative generation.

Genre confounding needs an explicit test. Train a simple classifier to predict `source_dataset` or `view_type` from text. If SetConCA gains track genre labels more than latent keys, the dataset is not yet clean.

Dynamic hard negatives should be used sparingly: for high-confidence entity/fact sets, ask a small generator to preserve entities and keywords while inverting the relation. These become adversarial candidates, not automatically trusted labels.

### Phase 3.5: Global split and decontamination

Add an explicit script:

```text
scripts/deduplicate_and_split_latent_data.py
```

It must run after all builders and before final manifests. It should:

- deduplicate by `source_dataset + source_id`;
- deduplicate by normalized text hash;
- check near-duplicates with MinHash or embedding similarity;
- prevent the same latent key from appearing in both steering-train and benchmark-eval when the claim requires source-disjoint generalization;
- write a split manifest explaining every exclusion.

This is necessary because contamination and benchmark leakage are now a known evaluation risk, especially when public datasets are reused across training, validation, and benchmark tasks.

### Phase 3.6: Data ethics appendix

Before adding RealToxicityPrompts, BBQ, or harmful-request refusal data, write:

```text
docs/DATASET_ETHICS_APPENDIX.md
```

It should state:

- which harmful or sensitive content is stored,
- whether raw text or only references/ids are stored,
- access controls and sharing policy,
- intended use,
- disallowed use,
- whether the data is train, dev, or eval-only,
- annotator/reviewer exposure risk,
- how results will report bias/toxicity findings without reproducing unnecessary harmful text.

For the first implementation, keep toxicity and bias sources eval-only or defer them entirely.

Sampling policy:
